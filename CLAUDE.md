## コミットメッセージ規約                                                      
                                                                                 
  [Conventional Commits](https://www.conventionalcommits.org/) に準拠する。      
                                                                                 
  | プレフィックス                                              | 意味               | 使う場面                                           |
  | ----------------------------------------------------------- | ------------------ | -------------------------------------------------- |
  | `feat:`                                                     | 新機能の追加       | 新しいページ、コンポーネント、データ項目の追加など |
  | `style:`                                                    | 見た目の変更       |
  | CSS修正、カラー変更、レイアウト調整など（機能は変わらない） |
  | `refactor:`                                                 | コード構造の改善   |
  | コンポーネント化、関数の整理など（動作は変わらない）        |
  | `fix:`                                                      | バグ修正           | 表示崩れ、リンク切れ、型エラーの修正など           |
  | `docs:`                                                     | ドキュメントの変更 | README、CLAUDE.md、コメントの追加・修正            |
  | `chore:`                                                    | 雑務               | 依存パッケージの更新、設定ファイルの変更など       |

  ### ルール
  - メッセージは英語で短く書く
  - 「何をしたか」ではなく「何のためにしたか」を意識する
  - `git add` は変更したファイルを個別に指定する（`git add .` は避ける）

  ポイント：

  - 表形式 — ぱっと見で引ける。Markdownのテーブルはこの書き方
  - 使う場面の具体例 — このプロジェクトに合わせた例にすると実感が湧く
  - ルールセクション — 今まで学んだことのまとめにもなる
